# Wordloom Compliance Documents Index

Status: Internal index / draft for review

Last reviewed: 5 May 2026

## Purpose

This index helps future chats, Codex, Wordloom support, school review conversations and internal planning understand the Wordloom compliance/support pack without opening every document first.

It explains what each document is for, who sees it, when it should be used and what it must not be used to claim.

This index is not a public policy page, legal advice, certification, compliance approval or confirmation that Wordloom is fully compliant.

## How to use this index

- Use it to find the right document quickly.
- Use it before school, DPO or procurement conversations.
- Use it before changing public claims.
- Use it before pilot or release readiness reviews.
- Use it when handing over to a new chat, developer or support person.
- Do not treat the existence of a document as proof that every action inside it has been completed.

## Document audience layers

### Public policy pages

Public policy pages are public-facing transparency pages. They can be read by schools, parents/carers, staff, DPOs, procurement reviewers and visitors.

These pages should stay polished, cautious and evidence-aligned. They should explain what Wordloom currently supports, what remains the school's responsibility and what limits apply.

### Internal compliance/support docs

Internal compliance/support docs are internal working documents. They are mainly for Wordloom, future developers/support, compliance review and school/DPO/procurement conversations where useful.

Some internal documents may be shared selectively with schools or DPOs where appropriate, but they are not public marketing pages and should not be presented as final legal approvals.

### Operational school/pilot docs

Operational school/pilot docs are practical working checklists for onboarding, support, pilot readiness, data requests and staff guidance.

They are mainly used by Wordloom/support during real school setup or pilot work. Some may be shared with schools where useful, but they remain draft/support documents unless formally approved.

## Quick "which doc do I need?" guide

- Public transparency wording -> public policy pages.
- Evidence behind public claims -> compliance evidence matrix.
- Draft contractual/data processing terms -> DPA draft.
- Data mapping -> records/data inventory.
- Supplier review -> supplier due diligence checklist.
- AI assurance -> AI risk notes and AI Use Statement.
- Teacher AI prompting rules -> teacher AI data minimisation guidance.
- Accessibility review -> Accessibility Statement and accessibility findings log.
- Incident handling -> incident response process.
- Export/deletion/offboarding -> retention/deletion/export procedure and school onboarding/offboarding checklist.
- Data subject requests -> data subject request handling checklist.
- School pilot readiness -> school pilot readiness checklist.
- Release readiness -> compliance release checklist.

## General wording guardrails

Avoid phrases such as:

- fully GDPR compliant
- complete compliance
- legally approved
- DPO approved
- procurement approved
- certified secure
- fully accessible
- WCAG compliant
- AI is risk-free
- completely safe AI
- all AI content is teacher approved
- all data can be instantly deleted
- fully automated export/deletion
- UK-only hosting
- no international transfers

Safer wording includes:

- designed to support
- currently evidenced by
- to be confirmed
- may
- where used
- where supported
- internal draft for review
- supports school/DPO/legal/procurement review
- not a substitute for formal review
